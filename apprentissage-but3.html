Contexte et démarche

Ce portfolio d'apprentissage documente ma progression au cours de ma 3ème année de BUT Informatique, parcours Intégration d'Applications et Management des Systèmes d'Information (IAMSI). Contrairement au portfolio de 2ème année centré sur l'optimisation, l'administration système et la gestion des données, cette troisième année marque un tournant vers la maîtrise professionnelle complète : architectures applicatives avancées, évolution de systèmes en production réelle, et conduite stratégique de projets informatiques.

Les compétences présentées ici s'appuient sur une double expérience complémentaire : celle de projets académiques rigoureux menés en équipe, et celle d'une alternance chez Ivalua, éditeur SaaS de rang mondial spécialisé dans la gestion des achats. Cette immersion professionnelle a considérablement enrichi ma compréhension des enjeux réels du développement logiciel, de la maintenance de systèmes complexes, et de l'intégration dans des environnements clients exigeants. Elle s'articule avec un stage chez CornerHub et plusieurs projets personnels significatifs comme Komuko.

Chaque apprentissage critique présenté est illustré par des réalisations concrètes et contextualisées, démontrant une progression mesurable d'une posture étudiante vers une posture professionnelle confirmée.

AC11

Choisir et implémenter les architectures adaptées

Contexte d'apprentissage

Tout au long de ma formation, j'ai été amené à concevoir et structurer des applications en choisissant les ressources techniques appropriées selon les contraintes et les besoins de chaque projet. Cette compétence repose sur une démarche systématique : analyser le besoin fonctionnel avant de sélectionner toute technologie, afin que chaque choix architectural soit justifié par ce qu'il apporte concrètement au projet.

Réalisations concrètes

  • The Paw-fice — Architecture MVC avec Symfony 6.4 LTS : L'application devait couvrir gestion d'utilisateurs, upload de fichiers, profils publics/privés et un panel d'administration dans un délai contraint. Ces exigences nous ont orientés naturellement vers Symfony 6.4 LTS et son architecture MVC, couplé à Doctrine ORM pour la persistance et Twig pour le rendu. J'ai pris en charge la partie Backend Features : conception de l'entité Animal avec génération automatique d'un hash SHA-256 à partir de l'email, implémentation des controllers AnimalController, AvatarController et ProfileController, et gestion de l'upload de fichiers avec BinaryFileResponse incluant une image par défaut selon le type d'animal. Chaque décision reposait sur une analyse précise du besoin fonctionnel avant toute ligne de code.
  • RateMySlide — Architecture API REST avec API Platform 4.2 : À deux sur le back-end, nous avons construit une API REST de critiques de toboggans avec Symfony 6.4 et API Platform 4.2. La décision structurante a été de choisir une architecture orientée API REST plutôt qu'une application monolithique classique, permettant un découplage total entre front-end et back-end. Pour l'authentification, nous avons intégré un système JWT via Lexik et Gesdinet Refresh Token, adapté aux API sans état. La conteneurisation via Docker garantit la portabilité de l'environnement. L'architecture interne est organisée en couches strictement séparées : entités, repositories, State Processors pour la logique métier, Voters pour les autorisations par ressource, et un EventSubscriber pour le rate limiting anti-bruteforce. La documentation OpenAPI générée automatiquement via API Platform rendait l'API immédiatement exploitable par le front-end sans effort de documentation manuelle.
Symfony 6.4 LTS API Platform 4.2 Doctrine ORM Twig JWT / Lexik Docker OpenAPI Architecture MVC REST API SHA-256
AC12

Faire évoluer une application existante

Contexte d'apprentissage

Faire évoluer une application existante implique une posture radicalement différente de la création from scratch : il faut lire, comprendre et intervenir sur du code que l'on n'a pas écrit, sans tout remettre à plat. Cette compétence, amorcée sur des projets académiques, a trouvé son expression la plus significative dans mon alternance chez Ivalua, où chaque intervention porte sur un système de grande envergure en production.

Réalisations concrètes

  • Les Aventuriers du Rail — Reprise de code existant : Ce projet académique nous a demandé de reprendre le code existant d'un jeu de plateau pour le corriger, le compléter et le faire évoluer sans en être les auteurs originaux. Ce premier exercice de reprise de code m'a appris à lire une base existante, à en identifier les conventions et invariants, puis à m'y intégrer de manière cohérente — sans réécrire ce qui fonctionnait.
  • Alternance Ivalua — Maintenance corrective sur C#/.NET : La plateforme Ivalua est une application de grande envergure dont chaque client dispose de son propre projet. Mon rôle lorsqu'un ticket arrive est d'abord d'analyser le comportement anormal : je récupère la branche du projet concerné, je reproduis le problème, puis j'inspecte le code C#/.NET pour en identifier la cause. Une fois celle-ci identifiée, la démarche diverge selon la nature du code en cause. Si l'erreur provient du code spécifique au client, j'apporte directement la correction. Si le problème est localisé dans le code produit standard, je propose un patch correctif propre au projet du client — une correction ciblée, clairement documentée — et je crée en parallèle un ticket à destination de la team R&D avec mon analyse et le patch comme base de travail. Ce processus m'a appris à intervenir avec méthode et précaution sur du code existant, en veillant à la qualité et à la documentation de chaque intervention.
C# / .NET Debugging Code Review Patch correctif Documentation technique Ticketing Analyse d'impact
AC13

Intégrer des solutions dans un environnement de production

Contexte d'apprentissage

L'intégration en production ne se résume pas à déployer du code : c'est un acte technique réfléchi qui exige de comprendre l'écosystème dans lequel s'insère la solution, de maîtriser les risques, et d'assurer la continuité de service. Cette dimension, je l'ai travaillée à travers des contextes très différents — du projet académique à l'environnement professionnel grand compte.

Réalisations concrètes

  • Stage CornerHub — Intégration dans un SI existant : Ma mission ne consistait pas à construire une application isolée, mais à analyser l'architecture déjà en place, comprendre les outils utilisés par les équipes, puis développer des applications complémentaires s'y intégrant naturellement. J'ai notamment développé une solution d'import et de partage de documents à destination de l'équipe de communication, déployée sur la même plateforme que les autres applications existantes. Cette expérience m'a sensibilisé à la contrainte réelle de l'intégration : une solution doit s'insérer dans un écosystème existant sans le perturber.
  • Ivalua — Déploiement en production client grand compte : Lorsqu'une correction est validée, c'est moi qui gère son déploiement dans l'environnement du client. Intervenir directement en production sur la plateforme d'un client grand compte implique une rigueur particulière : chaque déploiement doit être maîtrisé, testé et documenté, car la moindre erreur a un impact immédiat sur l'activité du client. Cette responsabilité m'a appris à aborder l'intégration en production comme un acte technique réfléchi, et non comme une étape automatique.
  • Komuko — Intégration avec un backend-as-a-service : Sur mon projet personnel Komuko, le choix de Supabase comme BaaS m'a permis de disposer d'un environnement de production directement accessible — PostgreSQL, authentification, stockage — sans configurer d'infrastructure serveur. Ce choix m'a appris à distinguer ce que l'on gère soi-même de ce que l'on délègue à un service managé, une réflexion centrale en intégration moderne.
  • Pipeline GitLab CI / SonarQube — Intégration continue : Dans le cadre académique, j'ai travaillé sur la mise en place d'un pipeline GitLab CI couplé à SonarQube pour automatiser l'analyse de qualité du code à chaque push. Cette expérience m'a initié aux principes de l'intégration continue : automatiser les vérifications pour qu'aucun code défaillant ne soit intégré sans avoir été contrôlé, une pratique indispensable dans tout environnement de production sérieux.
GitLab CI SonarQube Supabase PostgreSQL Docker Déploiement CI/CD Intégration continue Backend as a Service

AC51

Mesurer les impacts économiques, sociétaux et technologiques d'un projet informatique

Contexte d'apprentissage

Mesurer les impacts d'un projet informatique est une dimension que j'estime fondamentale dans toute démarche de conception sérieuse. Un projet ne se résume pas à sa faisabilité technique : il s'inscrit dans un contexte économique, organisationnel et sociétal qui doit être analysé avec une démarche proactive, créative et critique.

Réalisations concrètes

  • HealthSip — Startup fictive, bouteille connectée : Ce projet académique m'a amené à produire une analyse complète allant bien au-delà de la dimension technique : analyse de marché, matrice SWOT, étude concurrentielle et projections financières sur trois ans. Cet exercice m'a obligé à raisonner sur la viabilité économique d'une solution numérique, à identifier ses forces et ses risques, et à communiquer ces résultats de manière structurée aux différents acteurs du projet. C'est en travaillant sur ces projections que j'ai compris qu'un projet informatique génère des impacts qui dépassent largement son périmètre technique : coûts de développement, modèle de revenus, positionnement concurrentiel, et valeur perçue par les utilisateurs finaux.
  • Démarche transversale en entreprise et en projets : De manière plus générale, je considère que cette capacité d'analyse doit être appliquée à tout projet avec un niveau d'investissement proportionnel à ses enjeux. Sur des projets moins structurés, cela se traduit par une réflexion minimale mais systématique sur le coût, l'utilité et l'impact de ce que l'on construit — une habitude que j'essaie de cultiver aussi bien en contexte académique qu'en entreprise.
Analyse de marché SWOT Business Plan Projections financières IoT Analyse concurrentielle
AC52

Savoir intégrer un projet informatique dans le système d'information d'une organisation

Contexte d'apprentissage

Comprendre comment un projet informatique s'insère dans un système d'information existant implique de saisir la logique modulaire d'un SI, ses conventions, et la manière dont une nouvelle solution peut y être greffée sans remettre en cause l'existant. Cette compétence s'est construite progressivement, d'abord en cours puis en situation professionnelle réelle.

Réalisations concrètes

  • Dolibarr — Prise en main d'un ERP open source : En cours, j'ai eu l'occasion de manipuler Dolibarr, ERP open source, ce qui m'a permis de comprendre concrètement ce qu'est un système d'information d'entreprise : un ensemble d'outils interconnectés couvrant des besoins métiers variés (contacts, facturation, suivi de projets). Travailler sur cet outil m'a sensibilisé à la logique modulaire d'un SI et à la manière dont on peut y greffer de nouvelles fonctionnalités en respectant les normes et conventions en vigueur.
  • Stage CornerHub — Intégration dans un SI professionnel réel : J'ai mis cette logique en pratique concrète. J'ai analysé l'architecture applicative déjà en place avant de développer de nouvelles solutions, notamment une application d'import et de partage de documents, intégrée directement sur la plateforme existante aux côtés des outils déjà utilisés par les équipes. Cette expérience m'a appris qu'intégrer un projet dans un SI ne signifie pas seulement que le code fonctionne, mais que la solution s'inscrit de manière cohérente dans les usages et les flux de l'organisation.
Dolibarr ERP Système d'information Architecture SI Intégration applicative Analyse de l'existant
AC53

Savoir adapter un système d'information

Contexte d'apprentissage

Adapter un système d'information, c'est être capable de faire évoluer les outils et les processus d'une organisation pour répondre à de nouveaux besoins, tout en maintenant la cohérence et la continuité du SI existant. C'est la dimension de ma mission chez Ivalua qui illustre le mieux cet apprentissage.

Réalisations concrètes

  • Alternance Ivalua — Adaptation de la plateforme SaaS aux besoins clients : La plateforme Ivalua est déployée chez des clients aux besoins très différents, et une grande partie de mon travail consiste à comprendre le contexte spécifique de chaque client pour adapter la solution à ses contraintes. Lorsqu'un client remonte un problème ou exprime un besoin d'évolution, j'analyse son environnement, je propose une solution adaptée, et je la déploie dans son système en veillant à ce qu'elle s'intègre sans perturber les processus existants. Cette approche m'a appris à adopter une démarche proactive et critique : il ne s'agit pas d'appliquer mécaniquement une correction, mais de comprendre pourquoi le système se comporte d'une certaine façon et comment l'adapter de manière pérenne.
  • Socle théorique — Architectures SI et bases de données : Les cours sur les systèmes d'information, les bases de données et les architectures SI m'ont fourni le socle nécessaire pour raisonner sur l'adaptation d'un SI : comprendre les dépendances entre composants, anticiper les effets de bord d'une modification, et respecter les règles et normes en vigueur dans chaque intervention.

AC54

Organiser et partager une veille numérique

Contexte d'apprentissage

La veille numérique est une compétence que j'ai travaillée de manière structurée dans le cadre d'un projet académique dont l'objectif était de mettre en place un système de veille informationnelle automatisé et qualitatif. L'enjeu n'était pas simplement de collecter des informations, mais de les organiser et de les diffuser de manière efficace, en inscrivant cette démarche au sein d'une équipe avec des outils partagés.

Réalisations concrètes

  • Système de veille automatisé — Discord & IA : Nous avons mis en place une chaîne automatisée reposant sur Discord comme plateforme centrale, avec des bots configurés pour surveiller nos sources thématiques et envoyer des notifications à chaque nouvelle publication. Le contenu collecté était ensuite traité via intelligence artificielle pour générer automatiquement un pré-texte synthétisant les informations, avant d'être publié sur notre forum. Cette approche nous a permis d'obtenir un flux de veille qualitatif sans y consacrer un temps excessif, en développant une communication efficace et collaborative autour des informations partagées.
  • Veille personnelle continue : Au-delà de ce projet, je maintiens personnellement un niveau de veille minimum sur les technologies que j'utilise, que ce soit pour mon alternance ou mes projets personnels, en m'assurant d'être toujours à jour sur les évolutions qui me concernent directement.
Discord Bots automatisés Intelligence artificielle Veille technologique Curation de contenu Travail collaboratif
AC55

Identifier les enjeux de l'économie de l'innovation numérique

Contexte d'apprentissage

L'économie de l'innovation numérique est un domaine que j'ai exploré à travers plusieurs cours et projets académiques. Comprendre ces enjeux, c'est saisir que l'innovation numérique ne se limite pas à une idée technologique : elle doit répondre à un marché, se différencier de la concurrence et générer de la valeur économique mesurable.

Réalisations concrètes

  • HealthSip — Analyse économique d'un produit numérique : Le projet HealthSip m'a permis de travailler concrètement sur les enjeux économiques et concurrentiels d'un produit numérique : analyse de marché, étude concurrentielle, modèle de revenus et projections financières sur trois ans. Cet exercice m'a fait comprendre que l'innovation numérique doit répondre à un marché, se différencier de la concurrence et générer de la valeur économique mesurable.
  • Komuko — Blue Ocean Strategy et positionnement différenciant : Pour le dossier d'innovation de Komuko, mon application de suivi de manga et manhwa, j'ai appliqué la Blue Ocean Strategy et une analyse SWOT pour identifier un positionnement différenciant sur un marché pourtant déjà occupé. Cet exercice m'a appris à chercher des espaces de marché inexploités plutôt que de concurrencer frontalement des acteurs établis.
  • Règlement européen sur l'IA et RGPD : Les cours sur ces réglementations m'ont sensibilisé à la dimension juridique et éthique de l'innovation numérique, en veillant au respect des contraintes juridiques qui encadrent le développement et le déploiement de solutions numériques aujourd'hui.
Blue Ocean Strategy SWOT Modèle de revenus RGPD AI Act Innovation numérique Analyse concurrentielle
AC56

Guider la conduite du changement informatique au sein d'une organisation

Contexte d'apprentissage

La conduite du changement est une réalité centrale du déploiement de solutions informatiques : un outil, aussi bon soit-il techniquement, n'est adopté que si l'on accompagne les utilisateurs dans la mise en œuvre des évolutions avec des ressources pédagogiques adaptées et une communication claire.

Réalisations concrètes

  • Projet Obsidian — Dispositif d'onboarding utilisateur : Ce projet académique centré sur Obsidian comme outil de gestion de connaissances avait pour objectif de concevoir un dispositif d'onboarding permettant à de nouveaux utilisateurs de s'approprier l'outil progressivement, avec un prototype Figma et une base de connaissances intégrée. Ce projet m'a confronté à une réalité centrale de la conduite du changement : les ressources pédagogiques et la communication sont aussi importantes que la qualité technique de la solution.
  • Ivalua — Chaîne de communication lors des déploiements : Lorsqu'une correction ou une évolution est déployée sur l'environnement d'un client, les utilisateurs finaux doivent comprendre ce qui change et pourquoi. C'est souvent à travers les échanges avec les Technical Account Managers que cette information est transmise et vulgarisée. Cette chaîne de communication m'a appris que guider le changement informatique implique autant de rigueur dans la transmission de l'information que dans la réalisation technique elle-même.

AC57

Accompagner le management de projet informatique

Contexte d'apprentissage

Le management de projet informatique est une compétence que j'ai développée progressivement à travers mes expériences en équipe. Sur des projets académiques, j'ai appris à organiser le travail collectif, à répartir les responsabilités et à maintenir une cohérence entre les différentes parties d'un projet, en inscrivant ma démarche au sein d'une équipe pluridisciplinaire. En alternance, cette dimension prend une forme plus professionnelle avec des processus structurés et des clients à satisfaire dans des délais contraints.

Réalisations concrètes

  • RateMySlide et The Paw-fice — Organisation en équipe académique : Sur ces projets, j'ai appris à organiser le travail collectif, à répartir les responsabilités et à maintenir une cohérence entre les différentes parties du projet. Chaque membre apportait des compétences complémentaires, ce qui m'a appris à coordonner les contributions sans créer de dépendances bloquantes — un fondement du management de projet agile.
  • Alternance Ivalua — Management opérationnel en équipe RUN : Mon travail s'inscrit dans une équipe RUN structurée, avec des processus de suivi de tickets, des priorités à gérer et des clients à satisfaire dans des délais contraints. J'ai appris à estimer la complexité d'une tâche, à communiquer sur son avancement et à escalader les problèmes au bon niveau — vers les TAMs ou vers la team R&D. Développer une communication efficace et collaborative avec tous ces interlocuteurs est devenu un réflexe quotidien, aussi bien à l'écrit dans les tickets en anglais qu'à l'oral dans les échanges internes.

Synthèse BUT 3

Cette troisième année marque le passage d'une posture d'apprenant à une posture de praticien confirmé. Les compétences acquises s'articulent autour d'une conviction : la maîtrise technique est indissociable de la capacité à l'inscrire dans un contexte humain, organisationnel et économique.

Architectures et production

De la conception d'une API REST avec API Platform à la maintenance corrective en production chez Ivalua, j'ai acquis une vision complète du cycle de vie d'une application — du choix architectural initial jusqu'au déploiement en environnement client réel.

Systèmes d'information

À travers l'intégration dans des SI existants (CornerHub, Ivalua, Dolibarr), j'ai appris qu'une solution informatique ne vit pas en isolation : elle s'insère dans un écosystème d'outils, de processus et d'acteurs qu'il faut comprendre avant d'intervenir.

Conduite et innovation

Des projections financières de HealthSip à la Blue Ocean Strategy de Komuko, j'ai développé une capacité à penser un produit numérique au-delà de sa dimension technique, en intégrant les enjeux économiques, concurrentiels et réglementaires.

Communication et management

L'alternance chez Ivalua m'a forgé une communication professionnelle quotidienne — tickets en anglais, échanges avec les TAMs, escalades vers la R&D — et une capacité à travailler dans des contraintes de délai et de qualité réelles.
